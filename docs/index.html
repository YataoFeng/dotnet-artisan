场景示例

决策者先对齐需求，再路由到具体技能执行。

1. 搭建订单管理系统

你：我需要一个订单管理系统
AI：什么样的订单？采购订单、工单还是顾客订单？
你：顾客下单。未来可能拆分订单和支付。
AI：建议模块化单体 + DDD：订单、支付、库存作为限界上下文。
你：可以，但团队没用过 DDD。
AI：先做战略设计 — Event Storming → 领域文档 → 战术实现。.NET 10 + PostgreSQL。
行为链：4 轮提问 → domain-analyst → Event Storming → 领域文档 → 并行路由到 api + testing + devops + tooling

2. 排查生产崩溃（OOM）

你：生产环境 OOM 了，这是 dump 文件
AI：加载 WinDbg（Windows）或 dotnet-dump（Linux/macOS）分析。
行为链：dotnet-debugging → !analyze -v → !dumpheap -stat 发现 850MB → !gcroot 定位根因

3. 记住项目约定

你：记住，用 TimeProvider 而不是 DateTime.Now
AI：已捕获规则：TimeProvider 构造函数注入优先。已存入 MEMORY.md。
行为链：learning-agent → 泛化规则 → 去重 → 写入 MEMORY.md

30 秒法则

这个仓库生成的每个项目，新 AI 必须在 30 秒内理解。不需要聊天历史。

✅ 解决方案在根目录
❌ 藏在子目录里
✅ 项目名用领域术语
❌ Core、Shared、Services
✅ 新文件顶部一句话说明
❌ 不写或瞎猜
✅ 显式构造函数依赖
❌ Service Locator
✅ WHY 注释解释非明显决策
❌ WHAT 注释重复代码
✅ 无通用名称
❌ Helper、Manager、Utils
