决策者在行动

实际使用这个仓库时的真实行为示例。决策者（dotnet-advisor）弥合你的话和正确行动之间的鸿沟。

示例 1："我需要一个订单管理系统"

你: "我需要一个订单管理系统"
决策者: "什么样的订单？采购订单？工单？顾客订单？"
你: "顾客订单。状态：待确认→已确认→已发货。"
决策者: "明白了。.NET 10 还是 8？PostgreSQL？生产还是原型？需要测试吗？"
你: ".NET 10，PostgreSQL，生产环境，需要。"
行为链: 意图检测 → 3 轮提问 → 捕获领域词汇表 → 选择架构 → 脚手架生成 OrderManagement.Api → EF Core + Minimal API + 认证 → 配置 Testcontainers → 每个文件遵循 30 秒法则

示例 2："我的应用在生产环境崩溃"

你: "我的应用一直崩溃——OutOfMemoryException"
决策者: "有 .dmp 文件吗？有的话我加载 WinDbg 分析。"
你: "有，C:\dumps\crash.dmp"
行为链: 意图检测 → 加载 dotnet-debugging → WinDbg 打开转储 → !analyze -v → !dumpheap -stat 发现 850MB byte[] → !gcroot 追溯到 ImageCache 单例 → 报告根因 + 修复方案：用 IMemoryCache + SizeLimit

示例 3："审查我的代码"

你: "审查这个 PR"
行为链: 意图检测 → 启动 dotnet-code-review-agent（独立子会话）→ 分析正确性、性能、安全、架构 → 报告: "3 个问题：[严重] Scoped→Singleton DI，[性能] N+1 查询，[安全] 硬编码 API 密钥"

核心原则

KISS 优先

简单 CRUD 不需要 DDD。按问题规模选择模式。

禁止仓储包装

DbContext 就是工作单元；DbSet 就是仓储。

免费 & 开源

微软推荐 + MIT 协议。零商业依赖。

自文档化代码

新 AI 30 秒理解任何项目。指南 →

30 秒法则

这个仓库生成的每个项目，新 AI 会话必须在 30 秒内理解它。不需要聊天历史。

解决方案在根目录
绝不藏在子目录里
项目名用领域术语：OrderManagement.Api
绝不用 Core、Shared、Services
文件顶部一句话：处理履约：验证支付、预留库存、创建发货单
绝不写"订单处理"
显式依赖：构造函数(DbContext, IHybridCache)
绝不用 Service Locator、IServiceProvider
WHY 注释：IServiceScopeFactory 因为 BackgroundService 活得比 Scoped 长
绝不写"保存订单"
无通用名称
绝不出现 Helper、Manager、Utils、Common、Shared

发现模式

1. 打开 .slnx → 看到所有项目
2. 打开 Program.cs → 看到 DI、中间件、端点
3. 打开任意 .cs 文件 → 类名 + 构造函数 = 完整的故事
4. 打开 appsettings.json → 看到配置全貌

总计：30 秒。完整上下文恢复。零历史依赖。
